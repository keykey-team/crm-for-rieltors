# Регламент фронтенд-архитектуры (FSD)

> Источник истины для всех PR в этот репозиторий.
> Любой новый или изменяемый код на клиенте (`client/`) ДОЛЖЕН соответствовать этому документу.
> Расхождения — только по согласованию с тимлидом.

# Цель

- установить единые правила архитектуры фронтенд-кода в команде
- обеспечить предсказуемую структуру проекта, удобство поддержки, масштабируемость и безопасный рефакторинг

---

# 1. Архитектурные принципы и ограничения

> Архитектура — это набор ограничений, а не рекомендаций.

## 1.1. Общие глобальные правила архитектуры

1. зависимости разрешены только «сверху вниз»
2. запрещены циклические зависимости на любом уровне
3. любое архитектурное исключение должно быть осознанным и согласованным
4. путь к файлу должен объяснять его назначение без чтения кода
5. один файл — одна зона ответственности
6. дублирование логики запрещено
7. архитектура должна позволять безопасный рефакторинг
8. взаимодействие между частями системы происходит только через public API
9. временный код без явного назначения запрещён
10. запрещено держать `layout` как отдельный слой рядом со слоями FSD
    (`layout.tsx` — только в `app/`, UI-layout — через `widgets`)
11. слой `shared` и `app` — не имеет слайсов
12. page-section (локальная секция одной страницы), `widgets` — переиспользуемая секция приложения
13. допускаются архитектурные исключения при явной фиксации. **Согласовать с руководителем**

## 1.2. Ключевые изменения относительно прошлого подхода

1. сегмент `style` — **запрещён**. Все стили в `ui`
2. инфо-блоки, блоки без логики, которые находятся только на одной странице — относятся к слою `pages` (page-section, папка `ui` внутри слайса `pages`), а **не** `widgets`
3. все блоки, которые содержат домен / бизнес-сущность — относятся к слою `entities`, а **не** `widgets`
4. файлы имеют ограниченный размер, до 100 строк (исключения по согласованию)
5. папка `layout` **запрещена**, всё поведение описывается в слое `app`

# 2. Слои (layers)

> Слой — это архитектурный уровень ответственности.

## 2.1. Общие правила для слоёв

1. зависимости разрешены только сверху вниз
2. каждый слой используется строго по назначению
3. `shared` не зависит ни от одного слоя
4. `entities` не зависят от `features` и `widgets`
5. `features` не зависят от `widgets`
6. `pages` и `app` не содержат бизнес-логики
7. слой не используется как временное хранилище кода

## 2.2. Допустимые слои

### 2.2.1. `app` — инициализация приложения

| | |
|---|---|
| **Содержит** | `app/layout.tsx`, глобальные providers, глобальные стили, конфигурацию Next.js |
| **Используется для** | инициализации приложения, подключения инфраструктуры |
| **Импортировать можно** | `pages`, `widgets`, `features`, `entities`, `shared` |
| **Запрещено** | бизнес-логика, работа с доменными сущностями, реализация сценариев, создавать слайсы |

**Исключения для Next.js**
1. является инфраструктурным слоем роутинга и инициализации, а не FSD-слоем
2. допустимы только специальные файлы Next.js
3. запрещено создание FSD-слайсов

### 2.2.2. `pages` — страницы и роутинг

| | |
|---|---|
| **Содержит** | `page.tsx`, маршрутный `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`, секции страницы без логики |
| **Используется для** | связывания URL с интерфейсом, описания страниц, композиции из виджетов |
| **Импортировать можно** | `widgets`, `features`, `entities`, `shared` |
| **Запрещено** | бизнес-логика, пользовательские сценарии, прямые API-вызовы, сложная логика |

**Исключения для Next.js**
1. допустимы файлы роутинга (`index.tsx`, `[param].tsx`, `[...slug].tsx`, `_app.tsx`, `_document.tsx`, `_error.tsx`, `not-found.tsx`)
2. запрещено создание слайсов и сегментов в привычном понимании FSD
3. можно создать внутри `pages/<slice>` папку `ui` для секций страницы без логики

### 2.2.3. `widgets` — композиция и layout

| | |
|---|---|
| **Содержит** | крупные UI-компоненты, композицию `features` и `entities` |
| **Используется для** | самостоятельных секций интерфейса, свайперов, врапперов |
| **Импортировать можно** | `features`, `entities`, `shared` |
| **Запрещено** | пользовательские сценарии, бизнес-логика, прямые API-вызовы, работа со стором |

### 2.2.4. `features` — пользовательские действия

| | |
|---|---|
| **Содержит** | пользовательские сценарии, бизнес-логику, обработчики действий |
| **Используется для** | реализации действий пользователя, изменения состояния, взаимодействия с сервером |
| **Импортировать можно** | `entities`, `shared` |
| **Запрещено** | хранить доменные сущности |

### 2.2.5. `entities` — доменные сущности

| | |
|---|---|
| **Содержит** | доменные типы и модели, состояние, селекторы, инварианты, UI для одной сущности |
| **Используется для** | описания бизнес-объектов, отображения **одной** сущности |
| **Импортировать можно** | `shared` |
| **Запрещено** | пользовательские сценарии |

### 2.2.6. `shared` — переиспользуемый код

| | |
|---|---|
| **Содержит** | UI-примитивы, helpers и утилиты, общие типы, конфигурации и feature flags |
| **Используется для** | повторного использования, уменьшения дублирования |
| **Импортировать можно** | ничего |
| **Запрещено** | зависимости от других слоёв, бизнес-логика, доменные сущности, сценарии, создавать слайсы |

## 2.3. Запрещено

1. нарушать импорты снизу вверх
2. создавать циклические зависимости
3. определять дополнительные слои вне утверждённой архитектуры
4. смешивать ответственности слоёв
5. использовать слой не по назначению
6. дублировать один и тот же код в разных слоях
7. обходить public API слайсов между слоями

## 2.4. Ключевое правило

> Слой — это граница ответственности. Нарушение границы — архитектурная ошибка.

# 3. Слайсы (slices)

> Слайс — минимальная самостоятельная единица внутри слоя.

## 3.1. Общие правила

1. слайс всегда внутри слоя
2. один слайс = одна ответственность
3. у каждого слайса один public API
4. все внешние импорты — только через public API
5. внутренняя структура — деталь реализации
6. назначение слайса понятно из названия
7. `shared` и `app` — без слайсов

## 3.2. Допустимые слайсы

### 3.2.1. Pages-слайсы (кроме Next)
Описывает один экран приложения, существующий из-за URL.

Примеры: `home`, `catalog`, `product`, `profile`, `login`.

Запрещено: бизнес-логика, сценарии, доменные сущности, side-effects, хранение состояния.

### 3.2.2. Композиционные слайсы (widgets)
Самостоятельный UI-блок, но не действие и не сущность.

Примеры: `task-list`, `header`, `sidebar`, `footer`, `product-swiper`.

Запрещено: сценарии, бизнес-логика, side-effects, UI завязанный на одну сущность.

### 3.2.3. Слайсы действий (features)
Один пользовательский сценарий.

Примеры: `edit-task`, `delete-cart`, `login`, `add-to-wishlist`.

Запрещено: несколько сценариев в одном слайсе, доменные модели, layout, зависимость от `widgets`.

### 3.2.4. Доменные слайсы (entities)
Одна бизнес-сущность.

Примеры: `task`, `user`, `product`, `order`.

Запрещено: сценарии, side-effects, зависимость от `features`/`widgets`, UI более одной сущности.

## 3.3. Запрещено

1. слайсы без чёткой ответственности
2. несколько ответственностей в одном слайсе
3. зависимости между слайсами одного слоя
4. обходить `index.ts`
5. создавать слайсы в `shared` и `app`
6. импорты внутренних файлов слайса извне

## 3.4. Ключевое правило

> Слайс — это граница ответственности. Всё за её пределами недоступно по определению.

# 4. Сегменты (segments)

> Сегмент — тип файлов внутри слайса. Определяет роль кода, а не бизнес-назначение.

## 4.1. Общие правила

1. сегмент определяет техническую роль файла
2. каждый файл — в подходящем сегменте
3. сегменты создаются только при необходимости
4. допускается папка `common` в `ui` для UI, используемого в нескольких файлах того же `ui`

## 4.2. Допустимые сегменты

### 4.2.1. `ui` — только отображение
React-компоненты, стили, условный рендер, простая логика отображения (`if`, `map`).
Запрещено: бизнес-логика, API-вызовы, работа со стором, сложные вычисления.

### 4.2.2. `model` — доменная логика и состояние
Типы, интерфейсы домена, store, селекторы, инварианты, вычисляемые доменные функции.
Запрещено: технические утилиты, универсальные helpers без доменного смысла.

### 4.2.3. `lib` — вспомогательная логика
Pure functions, проверки, форматирование, вспомогательные вычисления (включая универсальные хуки).
Запрещено: знание конкретной сущности, владение сценарием, прямые API-вызовы.

### 4.2.4. `api` — работа с сервером
HTTP-запросы, функции запросов, DTO, технический маппинг.
Запрещено: бизнес-логика, стор, UI, склейка запросов в сценарий.

### 4.2.5. `config` — настройки
Константы, feature flags, статические настройки.
Запрещено: логика, side-effects, работа с данными.

## 4.3. Запрещено

1. смешивать ответственности сегментов в одном файле
2. нарушать назначение сегмента
3. импортировать внутренние файлы сегментов извне слайса
4. дублировать одинаковую логику
5. хранить временный код без явного назначения
6. создавать папки внутри сегмента, **кроме** `common` в `ui`
7. использовать недопустимые сегменты

## 4.4. Ключевое правило

> Сегмент отвечает на один технический вопрос.

# 5. Группировка слайсов

> Только по контексту использования. Группировка по техническому типу запрещена.

## 5.1. Контекст использования

Контекст — это ответ на:
1. где используется?
2. в рамках какого экрана, зоны интерфейса или сценария?

**Контекстом НЕ являются:**
1. тип UI (`modals`, `swipers`, `tabs`)
2. тип кода (`hooks`, `utils`, `api`)
3. техническая реализация

## 5.2. Когда группировать

1. в слое более 8 слайсов
2. структура становится нечитаемой
3. слайсы относятся к одному экрану / зоне / сценарию

## 5.3. Примеры

- **widgets:** `widgets/catalog/*`, `widgets/product/*`, `widgets/search/*`, `widgets/navigation/*`
- **features:** `features/auth/*`, `features/cart/*`, `features/product/*`, `features/orders/*`

## 5.4. Запрещено

1. группировка по типу UI
2. группировка по типу файлов
3. смешивание контекстов в одной группе

## 5.5. Ключевое правило

> Слайсы группируются только по контексту использования, а не по типу UI/реализации.

# 6. Public API

> Единственная разрешённая точка входа в слайс.

## 6.1. Общие правила

1. у каждого слайса один Public API: `<layer>/<slice>/index.ts`
2. любой импорт извне слайса — только через `index.ts`
3. Public API экспортирует только нужное снаружи
4. запрещён Public API на уровне сегментов
5. запрещены алиасы/пути, обходящие Public API
6. чтобы открыть новую часть — добавить в `index.ts`, а не импортировать по пути

## 6.2. Как импортировать

✅ Правильно:
```ts
import { TaskCard } from '@entities/task';
import { EditTaskButton } from '@features/edit-task';
import { Header } from '@widgets/header';
```

❌ Неправильно:
```ts
import { TaskCard } from 'entities/task/ui/TaskCard';
import { useEditTask } from 'features/edit-task/model/useEditTask';
import { Header } from 'widgets/header/ui/Header';
```

# 7. Правила именования

## 7.1. Слои, слайсы, сегменты

| Уровень | Формат | Примеры |
|---|---|---|
| Слой | `lowercase`, фиксированные | `app`, `pages`, `widgets`, `features`, `entities`, `shared` |
| Слайс | `kebab-case` | `task`, `edit-task`, `task-list`, `user-profile` |
| Сегмент | `lowercase`, фиксированные | `ui`, `model`, `lib`, `api`, `config` |

## 7.2. Файлы внутри сегментов

| Сегмент | Формат | Примеры |
|---|---|---|
| `ui` | `PascalCase.tsx` | `TaskCard.tsx`, `UserAvatar.tsx`, `Header.tsx` |
| `model` | `camelCase.ts` | `types.ts`, `useLogin.ts`, `slice.ts`, `invariants.ts` |
| `lib` | `camelCase.ts` | `formatDate.ts`, `isTaskOverdue.ts`, `buildQueryParams.ts` |
| `api` | `camelCase.ts` | `taskApi.ts`, `userApi.ts`, `authApi.ts` |
| `config` | `camelCase.ts` | `featureFlags.ts`, `routes.ts`, `taskConfig.ts` |

---

# 8. Ограничение на размер файлов

1. Максимум 100 строк, допустимая погрешность +20
2. Больше — только по согласованию с руководителем

---

## Чек-лист для PR

- [ ] Все импорты идут через public API (`@layer/slice`), без `slice/ui/...`
- [ ] Имена файлов соответствуют конвенциям (`PascalCase.tsx` для UI, `camelCase.ts` для остального)
- [ ] Слайсы в `kebab-case`
- [ ] Нет циклических зависимостей и нарушений направления импортов
- [ ] Нет файла > 120 строк без согласования
- [ ] Нет сегмента `style`, нет папки `layout`, нет лишних подпапок в сегментах (кроме `common` в `ui`)
- [ ] Доменные UI-блоки лежат в `entities`, page-only секции — в `pages/<slice>/ui`, переиспользуемые — в `widgets`
- [ ] Каждый новый слайс имеет `index.ts`, экспортирующий только нужное
