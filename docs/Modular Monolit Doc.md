# Регламент по Modular Monolith Architecture v1.0

# Содержание

# Цель

> установить единые правила архитектуры backend-кода в команде,
обеспечить предсказуемую структуру проекта, изоляцию модулей, безопасный рефакторинг, масштабируемость и возможность дальнейшего выделения модулей в отдельные сервисы.
> 

---

# *1.* Архитектурные принципы и ограничения

> Архитектура — это набор ограничений, а не рекомендаций.
> 

## *1.1.* Общие глобальные правила архитектуры

1. приложение остаётся единым монолитом
2. код разделяется на независимые бизнес-модули
3. каждый модуль имеет явные границы ответственности
4. зависимости между модулями запрещены напрямую
5. взаимодействие между модулями происходит только через публичный контракт
6. бизнес-логика не должна находиться в контроллерах
7. работа с базой данных не должна находиться в сервисах напрямую
8. инфраструктурный код не должен протекать в доменную логику
9. запрещены циклические зависимости на любом уровне
10. общий код не должен превращаться в свалку
11. любое архитектурное исключение должно быть осознанным и согласованным
12. путь к файлу должен объяснять его назначение без чтения кода
13. один файл — одна зона ответственности
14. временный код без явного назначения запрещён
15. архитектура должна позволять безопасный рефакторинг

## *1.2.* Ключевая идея

> Modular Monolith — это монолитное приложение, внутри которого код организован как набор независимых модулей.
Физически это один backend, один деплой и одна кодовая база.
Логически — это набор изолированных бизнес-контекстов.
> 

## *1.3.* Ключевые ограничения

1. модуль не импортирует внутренние файлы другого модуля
2. Public API модуля является единственной точкой входа
3. `common` используется только для универсального и инфраструктурного кода
4. бизнес-смысл всегда живёт внутри конкретного модуля
5. транзакция принадлежит сценарию, который меняет состояние системы

# *2.* Верхнеуровневая структура

> Верхнеуровневая зона определяет ответственность кода в backend-приложении.
> 

## *2.1.* Допустимые верхнеуровневые зоны

```
src/
├─ configuration
├─ common
├─ presentation
├─ modules
└─ tests
```

## *2.2.* Общая схема зависимостей

```
presentation -> modules
modules -> common
modules -> configuration
common -> ничего из modules
configuration -> ничего из modules
```

## *2.3.* Общие правила верхнего уровня

1. `presentation` отвечает только за внешний вход и запуск HTTP API
2. `modules` содержат бизнес-контексты приложения
3. `common` не знает о конкретных модулях
4. `configuration` не содержит бизнес-логику
5. `tests` не участвует в runtime-архитектуре приложения

# *3.* Архитектурные зоны

> Зона — это верхнеуровневая область ответственности внутри backend-приложения.
> 

## *3.1.* `configuration` — конфигурация приложения

> Отвечает за настройку приложения и подключение внешней среды.
> 

| **СОДЕРЖИТ** |   • env-конфигурацию
  • подключение к БД
  • настройки приложения
  • настройки внешних сервисов |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • запуска приложения
  • централизованного хранения конфигурации
  • подключения внешней среды |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `common`
  • `modules`
  • `presentation` |
| **ЗАПРЕЩЕНО** |   • бизнес-логика
  • HTTP-логика
  • логика конкретных модулей
  • сценарии пользователей |

**КЛЮЧЕВОЕ ПРАВИЛО**

> Configuration знает, как приложение подключается к внешней среде, но не знает, что приложение делает.
> 

## *3.2.* `common` — общесистемный код

> Содержит общесистемный код, который не принадлежит конкретному бизнес-модулю.
> 

| **СОДЕРЖИТ** |   • инфраструктурный код
  • минимальный shared kernel
  • общие технические механизмы
  • общие типы ошибок и permissions |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • переиспользования общесистемных механизмов
  • синхронизации минимального общего бизнес-языка
  • технической поддержки приложения |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • ничего из `modules` |
| **ЗАПРЕЩЕНО** |   • логика конкретного модуля
  • модели конкретного модуля
  • складывать код “просто чтобы переиспользовать”
  • превращать common в dump-папку |

### *3.2.1.* Допустимые зоны `common`

```
common/
├─ infrastructure
└─ shared-kernel
```

### *3.2.2.* `common/infrastructure`

| **СОДЕРЖИТ** |   • middleware
  • logger
  • event bus
  • db plugins
  • error handler |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • технической поддержки приложения
  • подключения инфраструктурных механизмов |
| **ЗАПРЕЩЕНО** |   • бизнес-логика
  • правила конкретного модуля
  • доменные сущности |

### *3.2.3.* `common/shared-kernel`

| **СОДЕРЖИТ** |   • общие permissions
  • общие error types
  • базовые value objects
  • общие enum’ы |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • синхронизации общего бизнес-языка между несколькими модулями |
| **ЗАПРЕЩЕНО** |   • логика конкретного модуля
  • модели конкретного модуля
  • services
  • repositories |

**ГЛАВНОЕ ПРАВИЛО**

> Shared Kernel должен быть маленьким.
Если Shared Kernel растёт слишком быстро — архитектура теряет модульность.
> 

## *3.3.* `presentation` — внешний вход в приложение

> Отвечает за запуск HTTP API и подключение внешнего входа в систему.
> 

| **СОДЕРЖИТ** |   • Express app
  • server startup
  • route mounting
  • swagger / docs |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • запуска HTTP API
  • подключения middleware
  • подключения модулей
  • описания внешней точки входа |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `modules`
  • `common`
  • `configuration` |
| **ЗАПРЕЩЕНО** |   • бизнес-логика
  • работа с БД
  • логика конкретного сценария
  • принятие бизнес-решений |

**КЛЮЧЕВОЕ ПРАВИЛО**

> Presentation подключает приложение к внешнему миру, но не реализует поведение системы.
> 

## *3.4.* `modules` — бизнес-модули

> Module — это самостоятельная бизнес-область внутри приложения.
> 

| **СОДЕРЖИТ** |   • независимые бизнес-контексты
  • controllers
  • services
  • repositories
  • models
  • adapters / facades
  • public API модуля |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • изоляции бизнес-логики
  • разделения системы на автономные контексты
  • подготовки модулей к возможному выделению в сервисы |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `common`
  • `configuration`
  • публичные контракты других модулей |
| **ЗАПРЕЩЕНО** |   • импортировать внутренности другого модуля
  • знать детали реализации другого модуля
  • создавать циклические зависимости
  • раскрывать внутреннюю структуру наружу |

**КЛЮЧЕВОЕ ПРАВИЛО**

> Модуль должен быть потенциально готов к выделению в отдельный сервис.
>

# *4.* Модули

> Модуль — это граница бизнес-контекста.
> 

Если код относится к конкретному бизнес-смыслу — он должен находиться внутри соответствующего модуля.

## *4.1.* Общие правила модулей

1. один модуль = один бизнес-контекст
2. модуль должен быть автономным
3. модуль не должен импортировать внутренности другого модуля
4. модуль может открывать наружу только Public API
5. внутренняя структура модуля является приватной
6. модуль сам владеет своей бизнес-логикой
7. модуль сам владеет своими моделями и репозиториями
8. модуль не должен знать детали реализации других модулей
9. запрещены циклические зависимости между модулями
10. модуль должен быть потенциально готов к выделению в отдельный сервис

## *4.2.* Внутренняя структура модуля

```
modules/catalog/
├─ controllers
├─ services
├─ repositories
├─ models
├─ adapters
└─ index.js
```

## *4.3.* Запрещено

1. импортировать внутренние файлы другого модуля
2. выносить бизнес-логику модуля в `common`
3. раскрывать структуру БД через публичный контракт
4. превращать facade в копию service
5. смешивать HTTP, бизнес-логику и БД в одном файле

## *4.4.* Ключевое правило

> Модуль — это не папка.
Модуль — это изолированный бизнес-контекст с публичным контрактом.
> 

---

# *5.* Внутренние зоны модуля

> Внутренняя зона определяет техническую роль кода внутри конкретного бизнес-модуля.
> 

## *5.1.* `controllers` — HTTP-слой модуля

> Controllers — это HTTP-слой конкретного модуля.
> 

| **СОДЕРЖИТ** |   • routes
  • request validation
  • auth middleware
  • вызов service |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • обработки HTTP-запросов
  • проверки входных данных
  • передачи управления в service |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `services`
  • common middleware
  • shared-kernel |
| **ЗАПРЕЩЕНО** |   • бизнес-логика
  • работа с БД
  • сложные вычисления
  • вызовы repositories напрямую
  • принятие бизнес-решений |

**ГЛАВНОЕ ПРАВИЛО**

> Controller принимает запрос, проверяет его и передаёт управление в service.
Controller не принимает бизнес-решения.
> 

## *5.2.* `services` — бизнес-логика модуля

> Services содержат бизнес-логику модуля.
> 

| **СОДЕРЖИТ** |   • use cases
  • бизнес-правила
  • сценарии модуля
  • проверки и ограничения |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • реализации поведения системы
  • выполнения сценариев модуля
  • принятия бизнес-решений |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `repositories`
  • `adapters`
  • shared-kernel
  • common infrastructure |
| **ЗАПРЕЩЕНО** |   • HTTP-логика
  • прямой доступ к req/res
  • прямое использование моделей БД
  • логика другого модуля
  • хранить SQL/ORM-запросы напрямую |

**ГЛАВНОЕ ПРАВИЛО**

> Service отвечает на вопрос: что должна сделать система?
> 

## *5.3.* `repositories` — работа с данными

> Repositories отвечают за работу с данными.
> 

| **СОДЕРЖИТ** |   • запросы к БД
  • создание / обновление / поиск данных
  • инкапсуляцию ORM / ODM |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • изоляции базы данных от бизнес-логики
  • выполнения операций хранения данных |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • `models` |
| **ЗАПРЕЩЕНО** |   • бизнес-логика
  • HTTP-логика
  • вызовы других модулей
  • принятие бизнес-решений |

**ГЛАВНОЕ ПРАВИЛО**

> Repository знает, как сохранить данные.
Service знает, зачем это делать.
> 

## *5.4.* `models` — структура данных модуля

> Models описывают структуру данных модуля.
> 

| **СОДЕРЖИТ** |   • mongoose schemas
  • database models
  • индексы
  • технические хуки модели |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • описания хранения данных
  • настройки схем данных конкретного модуля |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • внутри репозитория текущего модуля |
| **ЗАПРЕЩЕНО** |   • HTTP-логика
  • сценарии пользователя
  • логика других модулей
  • бизнес-оркестрация |

## *5.5.* `adapters` / Facades — публичный контракт модуля

> Adapter / Facade — это публичный контракт модуля для других модулей.
> 

| **СОДЕРЖИТ** |   • методы, доступные другим модулям
  • безопасные read/write операции
  • контракты взаимодействия |
| --- | --- |
| **ИСПОЛЬЗУЕТСЯ ДЛЯ** |   • общения между модулями без доступа к внутренностям
  • ограничения внешнего доступа к модулю
  • защиты внутренней модели модуля |
| **ИМПОРТИРОВАТЬ МОЖНО** |   • service текущего модуля
  • типы публичного контракта |
| **ЗАПРЕЩЕНО** |   • отдавать наружу внутренние модели
  • раскрывать структуру БД
  • превращаться в копию service
  • давать доступ к приватной реализации |

**ГЛАВНОЕ ПРАВИЛО**

> Facade показывает другим модулям только то, что им разрешено знать.
>

# *6.* Public API модуля

> Public API — единственная разрешённая точка входа в модуль.
> 

## *6.1.* Общие правила Public API

1. у каждого модуля есть `index.js`
2. всё, что не экспортировано из `index.js`, считается приватным
3. внешние импорты из модуля разрешены только через `index.js`
4. запрещено импортировать внутренние файлы другого модуля
5. Public API должен быть минимальным
6. Public API не должен раскрывать детали реализации
7. если нужно открыть новый контракт — он добавляется в `index.js`

## *6.2.* Как импортировать

**ПРАВИЛЬНО:**

```tsx
import { catalogFacade } from '@modules/catalog';
import { orderFacade } from '@modules/order-management';
```

**НЕ ПРАВИЛЬНО:**

```tsx
import { CatalogService } from '@modules/catalog/services/CatalogService';
import { ProductModel } from '@modules/catalog/models/ProductModel';
import { createProduct } from '@modules/catalog/repositories/productRepository';
```

## *6.3.* Ключевое правило

> Если код не экспортирован из Public API — его не существует для других модулей.
>

# *7.* Взаимодействие между модулями

> Модули не общаются друг с другом напрямую через внутренние файлы.
> 

## *7.1.* Допустимые способы взаимодействия

1. через Facade
2. через Domain Events
3. через Shared Kernel
4. через Application-level orchestration

## *7.2.* Facade

> Используется, когда одному модулю нужен безопасный публичный метод другого модуля.
> 
1. facade не раскрывает внутренние модели
2. facade возвращает только публичный контракт
3. facade не должен становиться копией service

## *7.3.* Domain Events

> Domain Event — это сообщение о том, что в системе уже произошло важное бизнес-событие.
> 

### *7.3.1.* Примеры

```
UserRegistered
OrderCreated
PaymentSucceeded
StockReserved
ProductArchived
```

### *7.3.2.* Правила

1. событие описывает факт, который уже произошёл
2. событие не должно требовать знания внутренней структуры модуля
3. обработчик события не должен создавать циклическую зависимость
4. события используются для слабой связанности модулей

## *7.4.* Shared Kernel

> Используется только для минимального общего бизнес-языка, который действительно нужен нескольким модулям.
> 
1. общие enum’ы допустимы
2. общие permissions допустимы
3. модели конкретного модуля запрещены
4. логика конкретного модуля запрещена

## *7.5.* Application-level orchestration

> Используется, когда сценарий координирует несколько модулей на уровне приложения.
> 
1. оркестрация не должна ломать границы модулей
2. модули всё равно вызываются через публичные контракты
3. бизнес-логика конкретного модуля остаётся внутри этого модуля

---

# *8.* Transaction Boundaries

> Транзакция должна принадлежать тому сценарию, который меняет состояние системы.
> 

## *8.1.* Общие правила транзакций

1. транзакция открывается на уровне сценария изменения состояния
2. repository не принимает бизнес-решение о транзакции
3. controller не управляет транзакцией
4. граница транзакции должна быть понятна из use case
5. межмодульная транзакция допускается только как осознанное архитектурное решение

## *8.2.* Ключевое правило

> Транзакция отвечает за целостность сценария, а не за удобство работы с БД.
>

# *9.* Правила именования файлов и папок

## *9.1.* Именование папок

| Архитектурная единица | Формат именования | Примеры |
| --- | --- | --- |
| **Верхняя зона** | `lowercase` | `modules`, `common`, `presentation`, `configuration` |
| **Module** | `kebab-case` или `lowercase` | `iam`, `catalog`, `order-management` |
| **Segment** | `lowercase` | `controllers`, `services`, `repositories`, `models`, `adapters` |

## *9.2.* Общие правила именования

1. название папки должно объяснять назначение без чтения кода
2. модуль называется по бизнес-контексту, а не по технической реализации
3. сегмент называется только допустимым фиксированным именем
4. временные и абстрактные названия запрещены

---

# *10.* Ограничение на размер файлов

1. максимум 100 строк
2. допустимая погрешность: +20 строк
3. больше — только по согласованию с руководителем

---

# *11.* Запрещено

1. импортировать внутренние файлы другого модуля
2. хранить бизнес-логику в controllers
3. хранить запросы к БД в services
4. использовать Common как dump-папку
5. создавать циклические зависимости

# *12.* Ключевое правило архитектуры

> Modular Monolith — это не “папки по модулям”.
Это строгие границы ответственности, публичные контракты и запрет на неявные зависимости.
Если границы не соблюдаются — это обычный монолит.
