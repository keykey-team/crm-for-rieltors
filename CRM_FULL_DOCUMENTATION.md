# Realtor CRM — Полная документация (для PM и разработчиков)

Актуальность: составлено по коду репозитория на 2026-05-21.

## 1) Обзор продукта (PM / Sales / Operations)

CRM ориентирована на риелторов и агентства недвижимости. Основной контур: лиды -> сделки -> задачи/календарь -> аналитика и операционные инструменты.

Что система дает бизнесу:
- Единая точка управления воронкой от первого контакта до закрытия сделки.
- Прозрачность по работе команды (кто ведет лида, на каком этапе сделка, где риск по дедлайнам).
- Снижение потерь лидов за счет задач, напоминаний, распределения и стандартов коммуникации.
- Базу знаний и шаблоны для единых стандартов продаж.

Для кого система:
- Руководитель отдела продаж / директор агентства.
- РОП / тимлид.
- Агент по недвижимости.
- PM/операционный менеджер, который настраивает процессы.
- Sales/Account/Customer Success для демонстраций и внедрения клиентам.

Ключевые разделы продукта:
- `Dashboard` — сводные KPI, последние лиды, воронка, задачи.
- `Leads` — список, фильтры, массовые операции, импорт.
- `Deals` — воронка и карточка сделки (комментарии, чеклист, поля).
- `Properties` — база объектов, включая юниты (`PropertyUnit` / шахматка).
- `Tasks` — постановка и трекинг задач.
- `Calendar` — события + ICS-подписка.
- `Settings` — профиль, бренд, пользователи, справочники, этапы воронки, правила распределения.
- `Tools` — чат, шаблоны, база знаний, автоматизации, логи действий, helper/assistant.

Поддерживаемые роли (фиксируем как текущую реализацию):
- `admin` — полный доступ.
- `director` — расширенный доступ (в т.ч. аналитика/админ-функции).
- `agent` — ограниченный доступ по назначению и/или кастомным permission.

Плановые ограничения функций (feature-gating):
- `free`: базовый набор + helper.
- `pro`: analytics, automations, templates, knowledge base, activity log, branding и др.
- `business`: все из `pro` + chat/team/distribution/aftercare.

## 2) Большая продуктовая картина (для PM и Sales)

### 2.1 Что именно автоматизирует CRM

1. Привлечение и обработку входящего потока лидов:
- ручное добавление;
- импорт (табличные данные);
- первичная квалификация и назначение ответственного.

2. Продвижение по этапам сделки:
- визуальная воронка;
- фиксация стоимости, комиссии и статуса;
- привязка объекта и истории взаимодействий.

3. Операционную дисциплину:
- персональные задачи;
- календарь встреч/показов;
- уведомления о рисках (просрочка, новые лиды, активность).

4. Командные стандарты:
- база знаний;
- шаблоны коммуникаций;
- кастомные поля и словари;
- аудит изменений.

### 2.2 Простое объяснение “о чем система”

Если коротко: это рабочая CRM для риелторской команды, где:
- лид не “теряется” после первого контакта;
- каждый этап сделки фиксируется;
- руководитель видит, что реально происходит в отделе;
- агент работает в одном месте, а не в 10 разных таблицах и мессенджерах.

### 2.3 Главные объекты системы (бизнес-уровень)

- `Lead`: потенциальный клиент и его запрос.
- `Deal`: коммерческий процесс с клиентом до результата.
- `Property`: объект недвижимости, с которым работает команда.
- `Task`: конкретное действие сотрудника с дедлайном.
- `Event`: событие календаря (встреча, показ, звонок).
- `Template` / `Knowledge Article`: стандарты работы и скрипты.

## 3) Подробно по блокам (что делает каждый модуль)

### 3.1 Dashboard (Дашборд)

Назначение:
- Быстрый операционный срез “что происходит сегодня”.

Что показывает:
- общее количество лидов/сделок/задач/объектов;
- новые лиды;
- активные сделки;
- задачи на сегодня;
- последние лиды;
- срез по этапам воронки.

Кому полезно:
- агенту: понять приоритеты на день;
- руководителю: увидеть узкие места по воронке и нагрузке.

### 3.2 Leads (Лиды)

Назначение:
- Управление входящим потоком и первичной квалификацией клиентов.

Функции:
- список лидов с фильтрами (`status`, `source`, `manager`);
- поиск;
- создание/редактирование/удаление;
- массовые действия (смена статуса, назначение менеджера, удаление);
- импорт лидов;
- автоназначение по правилам (`LeadDistributionRule`);
- создание сделки из лида.

Бизнес-эффект:
- скорость обработки входящих обращений;
- контроль конверсии “лид -> активная работа -> сделка”.

### 3.3 Deals (Сделки / Воронка)

Назначение:
- Управление сделкой на всех этапах.

Функции:
- Kanban-логика этапов;
- поля сделки (сумма, комиссия, валюта, заметки);
- привязка к лиду/объекту;
- комментарии;
- чеклист действий;
- кастомные поля для вашей методологии продаж.

Бизнес-эффект:
- понятный pipeline;
- предсказуемость выручки;
- контроль “зависших” сделок.

### 3.4 Properties (Объекты)

Назначение:
- Единый каталог недвижимости для работы отдела.

Функции:
- CRUD объектов;
- фильтры/поиск по типу, статусу и параметрам;
- хранение характеристик;
- `PropertyUnit` (шахматка/юниты) для пообъектной детализации.

Бизнес-эффект:
- ускорение подбора объектов под запрос;
- меньше ошибок и дублирования данных.

### 3.5 Tasks (Задачи)

Назначение:
- Операционная дисциплина и контроль исполнения.

Функции:
- создание задач по лидам/сделкам;
- типы задач (звонок, встреча, показ, документы и т.д.);
- приоритеты и статусы;
- дедлайны и признак просрочки.

Бизнес-эффект:
- меньше “забытых” действий;
- рост скорости реакции менеджеров.

### 3.6 Calendar (Календарь)

Назначение:
- Планирование встреч и синхронизация с внешними календарями.

Функции:
- события по месяцу;
- CRUD событий;
- генерация токена подписки;
- ICS-ссылка для Apple/Google/других календарей.

Бизнес-эффект:
- синхронная работа команды;
- снижение пропущенных встреч.

### 3.7 Analytics (Аналитика)

Назначение:
- Управленческая отчетность по воронке и эффективности.

Функции:
- расширенная аналитика за период;
- источники лидов;
- распределение по этапам;
- агрегаты по выручке/комиссии;
- активность агентов.

Бизнес-эффект:
- обоснованные решения по маркетингу и команде;
- понятная картина эффективности каналов.

### 3.8 Automations (Автоматизации)

Назначение:
- Шаблонизация рутинных сценариев по модели trigger -> action.

Функции:
- CRUD правил автоматизации;
- активация/деактивация;
- хранение параметров триггера/действия.

Бизнес-эффект:
- меньше ручных операций;
- стандартизированные реакции на типовые события.

### 3.9 Templates (Шаблоны)

Назначение:
- Быстрые, единообразные коммуникации с клиентами.

Функции:
- шаблоны сообщений;
- категории;
- переменные;
- хранение авторства.

Бизнес-эффект:
- быстрее ответы клиентам;
- единый tone of voice команды.

### 3.10 Knowledge Base (База знаний)

Назначение:
- Централизованное хранение регламентов, скриптов и best practices.

Функции:
- статьи с категориями и публикацией;
- поиск по заголовку/контенту.

Бизнес-эффект:
- ускорение адаптации новичков;
- сохранение экспертизы внутри компании.

### 3.11 Chat (Командный чат)

Назначение:
- Внутренняя коммуникация команды по рабочим контекстам.

Функции:
- комнаты (`direct/group`);
- сообщения по комнате;
- треды (`threadId`);
- mentions пользователей;
- управление комнатами (добавление/удаление участников, rename).

Бизнес-эффект:
- меньше потерь контекста;
- быстрые согласования внутри CRM.

### 3.12 Activity Log (Логи действий)

Назначение:
- Аудит и контроль изменений в системе.

Функции:
- журнал по сущностям и действиям;
- фильтрация по `entityType/entityId`;
- информация об инициаторе.

Бизнес-эффект:
- прозрачность и разбор спорных ситуаций.

### 3.13 Settings (Настройки)

Назначение:
- Конфигурация системы под бизнес-модель агентства.

Функции:
- профиль пользователя;
- бренд (название, логотип, цвет, тема);
- пользователи и роли;
- этапы воронки;
- custom fields сделок;
- словари;
- правила распределения лидов;
- aftercare-планы;
- тариф/план.

Бизнес-эффект:
- гибкая адаптация CRM под процессы конкретной команды.

## 4) Основные бизнес-процессы (PM)

1. Работа с лидами
- Создание лида вручную или импортом.
- Автораспределение по `LeadDistributionRule` (если пользователь не назначен явно).
- Массовые действия: смена статуса, назначение менеджера, удаление.
- Из лида можно создать сделку (`POST /leads/:id/create-deal`).

2. Воронка сделок
- Сделка содержит этап, сумму, комиссию, связанный лид/объект.
- В деталях доступны комментарии, чеклист и кастомные поля.
- Этапы воронки настраиваются в `Settings -> Funnel stages`.

3. Операционный контур
- Задачи с приоритетом/сроком, выборкой по статусам/типам.
- Календарные события + генерация токена и ICS-ссылки для внешнего календаря.
- Уведомления по просроченным задачам, активностям и новым лидам за день.

4. База знаний и шаблоны
- CRUD для шаблонов сообщений и статей базы знаний.
- Категоризация + публикация статей.

5. Командная и админ-работа
- Управление пользователями (admin/director).
- Настройки бренда (название, лого, цвет, визуальные параметры).
- Управление словарями, aftercare-планами, правилами распределения.

MVP-границы:
- MVP = весь функционал, который уже присутствует в кодовой базе.

## 5) Роли и логика доступа (для PM/руководителя)

`admin`:
- полный доступ ко всем разделам;
- управление пользователями и конфигурацией системы.

`director`:
- широкий доступ к данным и аналитике;
- управленческие функции сопоставимы с админскими в текущем коде.

`agent`:
- рабочие операции по назначенным сущностям;
- доступ ограничивается ролью, ownership и кастомными permissions.

Кастомные permissions:
- задаются массивом секций (`dashboard`, `leads`, `deals`, `properties` и т.д.);
- если permissions не заданы, у пользователя совместимость “полный доступ” (legacy-логика).

## 6) Пользовательские сценарии (для Sales/PM демонстраций)

Сценарий A: “Новый лид с рекламы”
1. Менеджер создает лида вручную или импортом.
2. CRM назначает ответственного по правилам распределения (или вручную).
3. Менеджер фиксирует квалификацию и статус.
4. По готовности создается сделка из лида.

Сценарий B: “Ведение сделки до закрытия”
1. Сделка появляется в воронке на стартовом этапе.
2. На каждом этапе фиксируются заметки, задачи, комментарии.
3. При необходимости привязывается объект.
4. Финальные этапы: `closed` / `rejected` / `cancelled`.

Сценарий C: “Контроль руководителя”
1. Открывается dashboard и analytics за период.
2. Проверяются узкие места: этапы с низкой конверсией, просрочки задач.
3. Настраиваются шаблоны, воронка, распределение и регламенты.

## 7) Ограничения доступа и безопасности (PM + Dev)

- Авторизация: `NextAuth` (frontend) + JWT cookie `crm_token` (backend).
- Middleware в клиенте ограничивает маршруты по role и `permissions`.
- На backend большая часть API под `authMiddleware` (cookie-based).
- Для `agent` действует ownership-фильтрация в ключевых сущностях (leads/deals/tasks/analytics и т.д.).

Важно: доступ к части CRUD проверяется только на frontend/ownership-логике конкретных endpoint’ов; централизованной RBAC-политики по всем маршрутам нет.

## 8) Что особенно важно для Sales

- Система продается как “операционная CRM для риелторов”, а не просто как таблица лидов.
- Ключевая ценность для клиента: прозрачность pipeline + контроль команды + стандарты работы.
- Типовые боли, которые закрывает CRM:
  - потеря лидов;
  - хаос в коммуникациях;
  - отсутствие понятной воронки;
  - слабая управляемость отдела;
  - сложность адаптации новых агентов.

Короткий pitch:
- “Мы даем вам единый рабочий контур: лиды, сделки, объекты, задачи, календарь и аналитику в одном интерфейсе, с настройкой под вашу структуру команды.”

## 9) KPI, которые можно отслеживать в системе

- Кол-во новых лидов за период.
- Конверсия по этапам воронки.
- Доля активных/потерянных лидов.
- Объем закрытых сделок.
- Комиссия и средний чек.
- Просроченные задачи.
- Активность менеджеров по лидам/сделкам/задачам.

## 10) Технологический стек (Dev)

Frontend (`/client`):
- Next.js 14 (App Router), React 18, TypeScript.
- NextAuth (Credentials provider).
- Tailwind + Radix UI + shadcn-style компоненты.
- Recharts, date-fns, framer-motion.

Backend (`/server`):
- Express + TypeScript.
- Prisma + PostgreSQL.
- JWT cookie auth (`crm_token`), bcrypt.
- AWS S3 presigned upload/download.

Инфраструктурно:
- Frontend проксирует `/api/*` на `http://localhost:4000/api/*` через `next.config.js` rewrite.

## 11) Архитектура репозитория (Dev)

- `client/app` — маршруты и layout’ы Next.js.
- `client/src/entities/*` — API-клиенты и типы по доменным сущностям.

- `server/src/modules/iam` — логин/сессия/регистрация.
- `server/src/modules/system` — основной бизнес-API.
- `server/prisma/schema.prisma` — модель данных.

## 12) Конфигурация окружения (Dev)

Backend обязательные переменные:
- `DATABASE_URL` (required)
- `JWT_SECRET` (required)
- `PORT` (default `4000`)
- `CLIENT_URL` (default `http://localhost:3000`)
- `AWS_REGION` (default `us-east-1`)
- `AWS_BUCKET_NAME` (optional, но нужен для файлов)
- `AWS_FOLDER_PREFIX` (optional)

Frontend:
- `NEXTAUTH_SECRET`
- `NEXTAUTH_URL` (для metadata и next-auth)
- `NEXT_PUBLIC_API_URL` (используется в login/signup запросах)
- `BACKEND_JWT_SECRET` или `JWT_SECRET` (для `/api/auth/crm-session`)

## 13) Локальный запуск (Dev)

1. Установить зависимости:
- `cd server && npm install`
- `cd client && npm install`

2. Backend:
- Настроить `DATABASE_URL`, `JWT_SECRET`.
- `npm run prisma:generate`
- применить миграции (в репозитории не вижу явного каталога миграций — проверить состояние БД отдельно).
- `npm run dev` (порт `4000`).

3. Frontend:
- Настроить `NEXTAUTH_SECRET`, `NEXTAUTH_URL`.
- `npm run dev` (порт `3000`).

## 14) Модель данных (Dev)

Ключевые сущности:
- `User`, `Lead`, `Deal`, `Property`, `PropertyPhoto`, `PropertyUnit`, `Task`, `Event`.
- `FunnelStage`, `DealCustomField`, `DealCustomFieldValue`.
- `Template`, `KnowledgeArticle`, `Automation`, `ActivityLog`, `Communication`.
- `LeadDistributionRule`, `AftercarePlan`, `AftercareStep`, `Dictionary`.
- `ChatRoom`, `ChatRoomMember`, `ChatMessage`, `ChatMention`, `HelperMessage`.

Особенности:

- Для части связей настроен `onDelete: Cascade`.

## 15) API-контракт (Dev, high-level)

IAM:
- `POST /api/iam/login`
- `POST /api/iam/signup`
- `GET /api/iam/session`
- `POST /api/iam/logout`

Core:
- Users, profile, brand, plan, permissions.
- Leads (CRUD, bulk, import, create-deal).
- Deals (CRUD, comments, checklist, custom fields).
- Properties + Property Units.
- Tasks.
- Events + calendar token + ICS feed.
- Analytics + dashboard stats.
- Templates, knowledge-base, automations.
- Dictionaries, funnel stages, aftercare plans, lead distribution.
- Notifications, global search, helper/assistant.
- Files (`/files`, `/upload/presigned`).

Практически все core-эндпоинты определены в `server/src/modules/system/routes.ts`.

## 16) Известные технические риски и расхождения (важно)

1. Chat-контракт:
- Каноничная модель: `rooms/threads` (совместимо с текущим frontend).
- Backend приведен к этой модели через `GET/POST /api/chat` + `PUT/DELETE /api/chat/rooms`.

2. Проверки прав неоднородны:
- Часть endpoint’ов строго учитывает ownership/role.
- Часть админских/операционных endpoint’ов полагается на frontend-ограничения.

3. Отсутствует тестовый контур:
- В репозитории фактически нет unit/integration/e2e тестов.

4. Exchange-rate endpoint зависит от внешнего NBU API:
- Есть кэш на 1 час и graceful fallback на stale данные, но при холодном старте без сети вернет 500.

5. Лимиты и пагинация:
- В ряде endpoint используются фиксированные `take` (200/300/500), но нет универсальной пагинации.

## 17) Что важно PM при планировании roadmap

- Приоритет №1: формализация и стабилизация чат-домена (rooms/threads) + покрытие тестами.
- Приоритет №2: единая серверная RBAC/ABAC политика вместо распределенных проверок.
- Приоритет №3: тестовое покрытие критических flows:
  - auth/session
  - lead -> deal
  - bulk operations
  - calendar token/ics
