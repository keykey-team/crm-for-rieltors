# AGENTS.md

Этот файл — обязательные инструкции для любого AI-агента (Copilot, Codex, и т.д.), создающего или изменяющего код в этом репозитории.

## 1. Архитектура клиента (обязательно)

Весь клиентский код в `client/` должен соответствовать регламенту **Feature-Sliced Design**, описанному в:

➡️ **[`docs/FRONTEND_ARCHITECTURE.md`](./docs/FRONTEND_ARCHITECTURE.md)**

Перед созданием/изменением любого файла на клиенте — прочитать этот документ.

### Краткая шпаргалка

- Слои сверху вниз: `app` → `pages` → `widgets` → `features` → `entities` → `shared`. Импорты только вниз.
- Слайсы — в `kebab-case` (`edit-task`, `create-showing`).
- Сегменты внутри слайса: `ui`, `model`, `lib`, `api`, `config` (только при необходимости).
- Файлы: `ui/*.tsx` — `PascalCase`, остальные — `camelCase.ts`.
- Public API — **только** `index.ts` слайса. Не импортировать `entities/task/ui/TaskCard` напрямую.
- Размер файла — до 100 строк (мягкий лимит 120).
- `shared` и `app` — без слайсов.
- Сегмент `style` запрещён, папка `layout` запрещена.
- Доменные UI-блоки → `entities`. Сценарии действий → `features`. Композиции → `widgets`. Page-only секции — внутри `pages/<slice>/ui`.

## 2. Архитектура сервера

`server/` — модульный монолит на NestJS, модули в `server/src/modules/<module-name>/` с подпапками `controllers`, `services`, `repositories`, `dto`. Следовать существующим модулям как образцу (`sales-pipeline`, `task-management`, `lead-management`).

## 3. Правила работы с PR

1. Все feature-PR таргетят ветку `feature/spark-adding-realtor-suite`, **не** `develop`.
2. Каждый PR — одна законченная фича.
3. Миграции Prisma — отдельной коммитом, с понятным именем.
4. i18n обязателен для всего нового UI: `uk`, `en`, `ru`. Украинский — основной, должен быть грамотным.
5. Тесты на бизнес-логику обязательны.
6. `ActivityLog` на create / update / delete доменных сущностей.

## 4. Параллельный рефакторинг

Если в процессе работы вы видите код, который **уже сейчас нарушает** регламент `FRONTEND_ARCHITECTURE.md` и при этом находится в файлах, которые вы и так трогаете для своей фичи — поправьте его в том же PR.

Если нарушение в файлах, которые вы НЕ трогаете для основной задачи — **не трогайте**. Сделать рефакторинг отдельной задачей. Не разрастать PR.

## 5. Чек-лист DoD (обязательно в описании PR)

- [ ] Соответствие FSD-регламенту
- [ ] Импорты через public API
- [ ] i18n uk/en/ru
- [ ] Тесты на ключевую логику
- [ ] Миграция (если есть изменения схемы) применяется чисто
- [ ] Lint / typecheck без ошибок
- [ ] Документация в `docs/`, если фича пользовательская

## 6. Что запрещено

- Менять корневой `tsconfig.json`/`next.config.*` без явной необходимости.
- Удалять существующие поля моделей Prisma без миграции и согласования.
- Мержить PR в `develop`. Это делает только владелец репозитория.
- Создавать новые слои или сегменты, не описанные в регламенте.
